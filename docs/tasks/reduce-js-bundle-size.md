# Reduce the frontend JS bundle size

_Status: investigation complete, not yet implemented. Created 2026-06-11._

## Problem

The production frontend bundle is too large:

```
build/dist/js/productionExecutable/
  klang-engine.<hash>.js        9.3 MB   ← main bundle (downloaded at initial load)
  235.<hash>.js                 864 KB   ← lazy chunk: pixi.js
  248.<hash>.js                 716 KB   ← lazy chunk: three.js
  klang-worklet.js              244 KB   ← audio worklet (separate)
  *.js.map                       ~21 MB  ← source maps (NOT downloaded by end users)
```

The **9.3 MB `klang-engine.*.js` is the initial-load payload** we want to shrink.

## What is actually in the 9.3 MB (source-map attribution)

Attributing generated bytes back to their source via the source map:

| Source bucket                                                    | Share     | ~Bytes       | Verdict                          |
|------------------------------------------------------------------|-----------|--------------|----------------------------------|
| **`sprudel/build` (KSP-generated)**                              | **36.8%** | **~2.4 MB**  | **Abnormal — the main offender** |
| `sprudel/src`                                                    | 10.7%     | ~0.7 MB      | expected (sprudel runtime)       |
| Kotlin stdlib (`compileSync/js`, `mnt/agent`, `buildAgent/work`) | ~17%      | ~1.1 MB      | expected baseline                |
| `@codemirror/*` + `@lezer/*` + helpers                           | ~8.6%     | **~0.57 MB** | the code editor (see §2)         |
| `@js-joda/core` (+ timezone)                                     | ~3.6%     | ~0.23 MB     | date lib                         |
| klangscript / klangblocks / audio_* / kraft / ultra / app `src`  | the rest  | —            | expected                         |

`pixi.js` (864 KB) and `three.js` (716 KB) are **already** in on-demand lazy chunks — they are **not** part of the
initial load and are not the problem.

## 1. Primary win — de-bloat the KSP-generated registration (~1.2–1.6 MB)

### Root cause

`sprudel/build/generated/ksp/.../GeneratedSprudelRegistration.kt` is **3.76 MB / 66,015 lines**, produced by
`KlangScriptProcessor`. It registers the whole KlangScript stdlib and emits **~1,578 raw closures** (1,215
`registerExtensionMethodWithSpecs` + 363 `registerFunctionWithSpecs`), each ~28 lines of **near-identical boilerplate**:

```kotlin
) { receiver, args, loc ->
    @Suppress("UNCHECKED_CAST")
    val typedReceiver = wrapAsRuntimeValue(receiver).convertToKotlin(SprudelPattern::class, loc)
    val callInfo = CallInfo(
        callLocation = loc,
        receiverLocation = (receiver as? StringValue)?.location ?: (receiver as? NumberValue)?.location,
        paramLocations = args.map { arg -> (arg as? StringValue)?.location ?: (arg as? NumberValue)?.location },
    )
    checkArgsSize(fn = "firstOf", args = args, expected = 2, location = loc)
    val n =
        convertArgToKotlin(fn = "firstOf", args = args, index = 0, cls = Any::class, nullable = false, loc = loc) as Any
    val transform = convertArgToKotlin(
        fn = "firstOf",
        args = args,
        index = 1,
        cls = Function1::class,
        nullable = false,
        loc = loc
    ) as ((SprudelPattern) -> SprudelPattern)
    wrapAsRuntimeValue(typedReceiver.firstOf(n = n, transform = transform, callInfo = callInfo))
}
```

`CallInfo` build ×1,578, `checkArgsSize` ×1,453, `convertArgToKotlin` ×1,817, the
`(x as? StringValue)?.location ?: (x as? NumberValue)?.location` idiom ×~1,578, function-name string literal ~5× per
closure. All of it is pure mechanical duplication; every registration is reachable so DCE keeps it; it minifies into ~
2.4 MB. (The closure *count* itself — function × receiver type, e.g. `firstOf` for `SprudelPattern`/`StringValue`/
`Function1` — is intended auto-lifting and is **not** changed; only the *per-closure size*.)

### Fix — move the boilerplate into shared runtime helpers; keep only the typed dispatch generated

The emitted `ParamSpec` list already fully describes each parameter (`name`, `kotlinType`, `isNullable`, `isVararg`,
`isOptional`, `default` — verified at `klangscript-ksp/.../KlangScriptProcessor.kt:1643-1652`), so the boilerplate can
be driven from the specs at runtime.

**a) Add runtime helpers** in `klangscript/src/commonMain/kotlin/runtime/NativeInterop.kt` (reuse existing
`checkArgsSize`, `convertArgToKotlin`, `wrapAsRuntimeValue`, `RuntimeValue.convertToKotlin`):

- `extractLocation(v: RuntimeValue): SourceLocation?` — the
  `(v as? StringValue)?.location ?: (v as? NumberValue)?.location` idiom, once.
- `buildCallInfo(receiver: Any?, args, loc): CallInfo` — uses `extractLocation`; `receiver = null` →
  `receiverLocation = null` (top-level).
- `convertArgs(fn, args, paramSpecs, loc): List<Any?>` — does
  `checkArgsSize(fn, args, paramSpecs.count { !it.isOptional }, loc)` then converts `min(args.size, paramSpecs.size)`
  non-vararg specs via `convertArgToKotlin(fn, args, i, spec.kotlinType, spec.isNullable, loc)`. Converting only the
  *present* args reproduces today's "convert optional only when supplied" arity behavior.

**b) Add thin registration wrappers** in `klangscript/src/commonMain/kotlin/builder/KlangScriptExtensionBuilder.kt`,
layered over the existing `registerExtensionMethodWithSpecs` / `registerFunctionWithSpecs`. Provide 4 variants (
extension vs top-level × with/without callInfo — the no-callInfo variant skips `buildCallInfo` to preserve today's
no-allocation path):

```kotlin
fun KlangScriptExtensionBuilder.registerExtMethodFast(
    receiver: KClass<*>, name: String, paramSpecs: List<ParamSpec>, receiverKotlinType: KClass<*>,
    dispatch: (receiver: Any, a: List<Any?>, callInfo: CallInfo) -> Any?,
) = registerExtensionMethodWithSpecs(receiver, name, paramSpecs) { rcv, args, loc ->
    val typed = wrapAsRuntimeValue(rcv).convertToKotlin(receiverKotlinType, loc)
    wrapAsRuntimeValue(dispatch(typed, convertArgs(name, args, paramSpecs, loc), buildCallInfo(rcv, args, loc)))
}
```

(Receiver conversion target differs from the dispatch key — `StringValue`→`String`, `Function1`→`PatternMapperFn` — so
pass `receiverKotlinType` explicitly; for `useConvertToKotlin = false` cases use a cast-only variant where `rcv` is
passed raw and cast in the dispatch lambda.)

**c) Rewrite the two verbose emitters** in `klangscript-ksp/src/main/kotlin/RegistrationModel.kt` —
`ArityDispatchItem.renderRegistration()` (lines 83–192) and `FileLevelExtItem.renderRegistration()` (lines 268–354) — to
emit a helper call plus a minimal dispatch lambda. No-defaults case becomes ~3 lines:

```kotlin
builder.registerExtMethodFast(cls, "firstOf", < specsExpr >, SprudelPattern::class) { r, a, callInfo ->
    (r as SprudelPattern).firstOf(
        n = a[0] as Any,
        transform = a[1] as ((SprudelPattern) -> SprudelPattern),
        callInfo = callInfo
    )
}
```

Has-defaults case keeps the existing arity `if (a.size >= n) … else …` chain **inside** the dispatch lambda, reading
from the pre-converted `a` and using `a.size` — no inline `convertArgToKotlin`/`CallInfo`/`checkArgsSize`. Reuse
`castSuffix` (applied to `a[i]`), `joinCallArgs`, `withCallInfo`. Per-arg cast types (distinct from `kotlinType`) stay
in the generated dispatch where the precise type is known. Leave `FixedMethodItem`, `VarargItem`, `RawArgsItem`,
`RawBlockItem` as-is.

### Behavior-preservation checklist

- Required count `specs.count { !isOptional }` must equal today's `count { !hasDefault }`.
- Per-arg `nullable` now from `spec.isNullable` (emitted, verified) instead of the inline arg.
- Error messages keep the script function name.
- `CallInfo` only built for functions that use it.
- Varargs untouched (they use `VarargItem`).
- Named-arg resolution unaffected (`paramSpecs` still passed through).

### Expected result

~28-line closures → ~3–6 lines; ~2.4 MB → ~0.8–1.2 MB. **Saves ~1.2–1.6 MB off the 9.3 MB main bundle** (~13–17%),
unconditional (this is always-needed stdlib dispatch, no UX tradeoff). Bonus: 66k → far fewer generated lines speeds
compilation.

## 2. CodeMirror async loading — answer & analysis (~0.55–0.65 MB)

**How much would loading CodeMirror async (like the Kraft addons) save off the initial bundle?**

**~550–650 KB** — the measured `@codemirror/*` (≈421 KB) + `@lezer/*`/`@marijn`/`style-mod`/`w3c-keyname` (≈147 KB)
footprint, ≈6–7% of the 9.3 MB. (These are npm packages that ship clean source maps, so attributed ≈ actual minified.)

**Why it's currently in the main bundle even though a dynamic loader exists:**
`klangscript-ui/.../codemirror/CodeMirrorModules.kt` already does `js("import('@codemirror/state')")` (the same
dynamic-import pattern the Kraft pixi/three addons use). BUT the external declarations in
`klangjs/src/jsMain/kotlin/codemirror/ext/*.kt` are annotated **`@file:JsModule("@codemirror/...")`** (9 files: State,
View, Commands, Language, LangJavascript, BasicSetup, Lint, Autocomplete, LezerHighlight). A `@file:JsModule` is a *
*static** import; webpack puts the module in the main bundle and the dynamic `import()` then just resolves to the
already-bundled copy. The split never happens.

The Kraft addons avoid exactly this — `ultra/kraft/addons/pixijs/.../js/pixi-definitions.kt` even notes _"The JS library
is loaded dynamically — no static @JsModule import"_ — which is why pixi/three land in lazy chunks (235/248).

**What it would take:** remove `@file:JsModule`/`@file:JsNonModule` from the 9 `klangjs/codemirror/ext/*.kt` files and
access every CodeMirror symbol through the dynamically-imported module objects returned by `CodeMirrorModules.load()`
instead of the static `external class` companions (`EditorState.create(...)` → `modules.state.EditorState.create(...)`,
etc.). That re-touches the consumers that use the static classes directly:
`klangscript-ui/.../codemirror/{CodeMirrorComp, CodeMirrorTheme, DslCompletionSource, DslEditorExtension}.kt` and
`src/jsMain/kotlin/codemirror/CodeMirrorHighlightBuffer.kt`, and any editor mount must `await` the load before
constructing the view.

**Caveat on the benefit:** the savings come off the *initial download*, but Klang is a live-coding app where the editor
is the primary surface shown on load — so the ~0.6 MB is needed almost immediately. Async loading therefore **defers**
rather than **eliminates** the cost: the app shell (header/controls) paints sooner and the main bundle parses faster,
but time-to-editable is roughly unchanged. It is a real win for first paint and for any route that *doesn't* show the
editor (e.g. a landing/credits page), a smaller and more conditional win than §1, and a meatier refactor (loses some
static type ergonomics).

## 3. Optional — source maps in the production dist (~21 MB on disk, 0 KB initial load)

`build.gradle.kts:53` sets `devtool = "source-map"` in `commonWebpackConfig`, which applies to production too, emitting
external `*.js.map` (16 MB for the engine alone). Browsers fetch these **only when devtools is open**, so they do **not
** affect end-user initial load — but they bloat the deploy. If deploy size matters, switch the production webpack to
`hidden-source-map` (or stop shipping `.map`). Low priority; no user-facing speedup.

---

## Recommended sequencing

1. **§1 generated-registration de-bloat** — biggest, unconditional win (~1.2–1.6 MB), self-contained (3 hand-written
   files + 2 processor emitters), no UX tradeoff. Do this first.
2. **§2 CodeMirror async** — ~0.6 MB, improves first paint; do if the extra split + refactor is worth it, especially
   valuable once there is a non-editor landing route.
3. **§3 source maps** — only if deploy size is a concern.

Combined §1 + §2 ≈ **1.8–2.2 MB off the initial 9.3 MB (~20–24%)**.
