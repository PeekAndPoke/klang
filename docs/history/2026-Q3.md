Roughly two acts. **First:** finish and polish the engine — the master / loudness stage and the
Engine DSL. **Then:** with a grown, polished engine under us, this becomes the **quarter of tutorials** —
building out new ones and, above all, fixing the current set.

## Multi-engine audio backend (finish what Q2 started)

The headline. Today the backend is one singleton — every playback mixes into one
`Cylinders` + one hardcoded master limiter, which already lets two songs on the same
orbit corrupt each other's FX tails.

- ✅ **DONE (Q2).** Per-`playbackId` **PlaybackEngine** behind a **PlaybackEngineDispatcher**;
  each engine owns its full render state (scheduler + cylinders + render context), no shared
  scratch → multi-thread ready. Plus `AudioBackendContext` + `BackendClock` (shared state + one
  timeline; fixed first-notes-lost).
- ⬜ **NEXT — the actual feature.** A thin **master / loudness stage** (Song → `Cmd.SetMaster` →
  per-engine output) to prove the wiring; richer master chain (glue/eq/drive/ceiling), **crossfade**,
  and metering follow.
- ✅ Mixdown fast path: 1 engine renders straight into the mix (the ≥2-engine shared scratch lands
  with the master stage).

## Resource warehouse pool (the alloc-spike killer)

- Self-balancing pool for expensive per-engine resources (the ~7.68 MB delay rings,
  cylinders). Keep N ready, warm up ahead, refill/shrink ±1 per render cycle.
- Kills the first-note allocation spike (the "Der Schmetterling" stutter) and caps memory.
- Deliberately scheduled **last** in the backend work, after the FE/BE cleanup.

## Engine DSL — finish + extend

- Application path: wire `.engine(EngineDsl)` end-to-end (the one remaining Phase-1 item).
- Phase 2: typed oscillator subtypes. Phase 3: `EngineDefault` profiles.
- Likely rename `EngineDsl` → `PipelineDsl` (it's really a per-voice pipeline; "engine"
  now belongs to the PlaybackEngine layer).

## The quarter of tutorials (act two)

Once the master stage and the Engine DSL are fully ready and polished, the rest of the
quarter goes here. The engine has grown a lot — the tutorials haven't kept up.

- **Fix the current set first.** Most of them have ugly-sounding examples and don't get
  to the *core* of what they're teaching.
- **Build a stringent through-line.** Right now the path from the basics (a plain saw)
  up to a fully crafted sound isn't coherent — each tutorial should be a clear step along
  that line, not a scattered demo.
- Then **build out** new tutorials on top of that spine, showing off the grown engine
  (Engine DSL, body/vowel, drift, the master stage).
- Lean on the music-writing / recording skills and the tutorial factory to keep the
  examples actually musical (sound first).

## FE/BE state placement

- ✅ **DONE (Q2).** Mirror backend granularity on the frontend (playbackId-bound vs global):
  custom-osc / engine state → per-playback (leak fixed), FE↔BE clock offset → `KlangPlayer`

- ⬜ Worklet **clock divergence** fix (worklet reports wall-time / re-anchor baseTimeMs) — deferred.
