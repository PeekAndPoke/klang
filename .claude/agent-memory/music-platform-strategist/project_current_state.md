**What exists (as of 2026-03-20):**

- Kotlin Multiplatform engine (web, standalone, embeddable)
- Web frontend: run built-in songs, create new songs
- Engine sounds good (positive audio quality feedback)
- Oscilloscope / waveform visualization (in progress)
- Signal processing: distortion, effects ongoing
- PlaybackSignals: outbound musical events driving real-time code highlighting (first incarnation of music-to-outside
  binding)

**Early feedback from friends/family:**

- Discoverability/onboarding is a wall for non-technical users
- Value is there once people get past initial confusion
- Sound quality impresses people

**Strategic identity (clarified 2026-03-20):**

- Klang is a **live music engine** -- music is code that runs, reacts, and emits events
- Music is NEVER recorded to files. Always generated live from Sprudel patterns at runtime.
- Two-way reactive binding is the core differentiator:
    - Inbound hooks: external events change the music (game state, sensors, user input)
    - Outbound signals: musical events change the world (PlaybackSignals -> code highlighting, and eventually game
      mechanics, visuals, lighting)
- NOT a DAW. NOT a learning platform. NOT a jukebox.
- Closest competitor: Reactional Music (but Reactional is a proprietary black box; Klang is transparent/code-first)
- Unique positioning: only system combining embeddable + generative + transparent + pattern language + two-way reactive

**Audience priority for early adoption:**

1. Creative coders / generative artists (comfortable with code, want embeddable music)
2. Web developers building interactive products
3. Game developers (long-term, high trust bar)
4. Live performers / experimental musicians
5. Curious learners (served by transparency, not separate features)

**Why:** Founder clarified that the long-term vision is music-as-runtime, not music-as-recording. This changes the
competitive frame from DAWs to interactive music middleware.

**How to apply:** Feature recommendations should serve the engine's capability as an embeddable, reactive music runtime.
The web frontend is the near-term product AND the showcase for the engine. Pedagogic value is emergent from
transparency, not a separate feature set.
